### §2 — Sacred-data redesign — **Built (2026-06-20)**
**Current:** AI may NEVER touch `user_provided` fields or `booked` items (hard block; silently skipped in proposals). **User wants:** AI *should* be able to change "soft" content (title, notes, category) on user/booked items **with explicit confirmation**; only "hard" booked facts (date, time, location, cost when `booked`) stay locked with a "change manually" hint.

**Proposed model — classify fields, don't blanket-block:**
- **HARD-locked** (never AI-writable): on a `booked` item → `date`, `startTime`, `endTime`, `originName/Lat/Lng`, `destinationName/Lat/Lng`, `costCents`. Diff shows these as "🔒 booked — change manually."
- **SOFT / confirm-required**: `title`, `notes`, `category`, and any `user_provided` field on a non-booked item. AI MAY propose changes — rendered as a normal selectable diff but with a ⚠️ "overrides your edit" / "edits a booked item's label" badge, **unchecked by default** so applying is a deliberate act.
- The existing preview-then-commit IS the confirmation mechanism — so this is mostly: split the guard from one boolean into `lockLevel(field, item) → "hard" | "confirm" | "open"`, stop auto-dropping soft proposals, and surface them as warn-badged opt-in diffs. The accept path keeps a hard guard for HARD fields only.
- **Transparency fix for #1:** when the model omits/blocks fields, the panel should say so ("4 notes left unchanged — they're yours; check to override"). Provenance must still flip correctly: a confirmed AI edit to a `user_provided` field re-stamps to `ai_assumption`? → **NO — keep it `user_provided`** (the human is approving on their own behalf), or add `user_confirmed_ai`? Open question, decide at design.
- ⚠️ This relaxes the PRD "AI must NEVER overwrite user_provided/Booked" keystone. Acceptable product evolution but update CLAUDE.md's keystone wording when built.
- **Shipped as:** `fieldLockLevel()`/`deleteLockLevel()` in `guard.ts`; planner `annotateActions` tags each action `lockLevel: open|confirm|blocked` + `hardReasons`/`confirmReasons`; accept route drops hard fields & applies confirm fields the user selected. Panel: confirm diffs are amber, **unchecked by default** (deliberate opt-in), with ⚠️ reasons; hard fields show 🔒 "change manually" even on otherwise-open actions (transparency for #1). **Open question resolved:** a confirmed AI edit to a `user_provided` field **keeps `user_provided`** (user curates it; protects from future silent AI overwrite). CLAUDE.md keystone wording updated. Tests: `copilot-guard` (classification, mutation-proven) + `copilot-flow` (confirm-applies + hard-drops + provenance, end-to-end).

## [S7-TZ-1] + [S7-TZ-2] Timezone-aware itinerary — Built 2026-06-24

Built together (the foundation alone has no user-visible effect). Full plan: `~/.claude/plans/under-map-improvements-there-quiet-porcupine.md`.

**Model.** `startTime` is wall-clock in the item's ORIGIN tz; `endTime` wall-clock in its DESTINATION tz; `durationMinutes` is the source of truth for ELAPSED time. All ordering/duration math runs on UTC INSTANTS. IANA names (DST-correct), never frozen offsets.

**Schema.** `itineraryItems.originTimezone` / `destinationTimezone` (derived cache, OUTSIDE provenance — like `routeGeometry`); `trips.homeTimezone` (calendar display axis + coord-less fallback). Nullable adds, applied via `db:push`.

**Deps.** `tz-lookup` (offline lat/lng→IANA, no key), `luxon` + `@types/luxon` (instant↔zoned math), `@fullcalendar/luxon3` (named-tz grid). Ambient type decl: `src/types/tz-lookup.d.ts`.

**Code.**
- `src/lib/trip-state/tz.ts` — the ONLY module touching Luxon: `localToInstant`, `instantToLocal`, `arrivalLocal`, `zoneAbbrev`, `itemOriginTz`/`itemLocalTz`/`isCrossTimezone`. Tests `tz.test.ts` (LAX→JFK, red-eye, DST spring-forward, same-tz reduction).
- TZ-1 derivation in `geocode/route.ts`: sets origin/dest tz from final coords (recompute when coords move OR backfill when null); lazily sets `trips.homeTimezone` from the earliest dated grounded stop. Re-running the geocode endpoint backfills existing trips.
- `sequence.ts` rewritten: cursor is a Luxon UTC instant; emits start in origin tz, end in dest tz; cursor flips zones automatically after a flight. Backward-compatible (same-tz → identical output). `ScheduleChange` gained optional `endDate`. Tests `sequence.test.ts`.
- `calendar-view.tsx`: luxon3 plugin + `timeZone={homeTz}`; events built as UTC instants (red-eye rolls past midnight natively); drop writeback converts instants back through origin/dest tz; local label rendered from `formatItemTimeLabel`.
- `format.ts`: `formatItemTimeLabel` (full `09:00 PST → 17:30 EST`, `+1` on rollover) + `formatItemTzBadge` (compact `→EST` for the table). Used by table (Start cell `adornment` on `EditableCell`), map popup (`trip-map.tsx`), calendar.

**Display decision.** Calendar grid runs in ONE display tz (home) for honest geometry; LABELS are local everywhere. Per-segment-local grid axis deferred (one FullCalendar timegrid can't change tz mid-axis).

**Backfill note (user action):** existing trips show tz only after the geocode endpoint runs once (the "Geocode" button) — that's when columns populate and `homeTimezone` is set.

## Motion pass — one shared animation vocabulary — Built 2026-06-25

(Was the **Motion** item under "Aesthetics & UX polish (Phase 4.5)": "smoother
transitions/animations (panel expand, diff apply, map fly, row changes). Currently
abrupt. P2.") Hybrid CSS-tokens + framer-motion approach (user-chosen), all four
surfaces in one pass.

**Foundation (shared vocabulary).**
- `globals.css`: motion tokens as CSS custom properties — `--motion-fast/base/slow`
  (120/200/320ms) + `--ease-standard/entrance/exit` cubic-beziers; `@keyframes`
  `fade-in` / `fade-slide-up`; `.animate-fade-in` / `.animate-fade-slide-up` /
  `.marker-pop` utilities; a `@media (prefers-reduced-motion: reduce)` block that
  zeroes the duration tokens and disables the keyframes.
- `src/lib/motion.ts` (new): the framer counterpart — `DURATION`/`EASE` (same
  numbers as CSS), `transitionBase`, and `Variants` presets `panelExpand` (height
  auto + opacity), `cardEnterExit`, `staggerChildren`, `rowEnterExit`. Single source
  of truth for the framer layer; CSS block is the source for the CSS layer — keep in sync.
- `providers.tsx`: wrapped the tree in `<MotionConfig reducedMotion="user">` so every
  framer animation auto-respects the OS "Reduce motion" setting (mirrors the CSS guard).

**Surfaces.**
1. **Panel expand** (`page.tsx`): Calendar collapsible panel now `AnimatePresence` +
   `motion.div` with `panelExpand` (animated height + opacity); chevron is a
   `motion.span` rotating 0↔90°. Right-panel tab content crossfades (`AnimatePresence
   mode="wait"` keyed by `activeRightTab`).
2. **Diff apply** (`planning-panel.tsx`): the "Proposed changes" block wrapped in
   `AnimatePresence` (fades + rises out on accept/dismiss so the diff visibly
   "applies"); the card list is a `motion.div` with `staggerChildren`; each
   `ActionCard` is a `motion.label` with `cardEnterExit` so cards cascade in.
3. **Map** (`trip-map.tsx`): markers get a `.marker-pop` class on creation —
   **opacity fade-in ONLY** (markers are diffed via `markersRef`, so it fires only for
   genuinely new markers). NOTE: marker pop must NOT animate `transform` — MapLibre
   positions the marker element via inline `transform`, and a CSS transform animation
   (esp. `both` fill) would override it and strand the marker at the map origin; so the
   `marker-pop` class maps to the `fade-in` keyframe. Routes fade in: the visible drive
   layer is added at `line-opacity: 0`, then a `requestAnimationFrame` ramps it to
   `ROUTE_BASE_OPACITY` (MapLibre transitions paint changes by default). Camera moves
   (`easeTo`/`flyTo`/`fitBounds`) were already eased — left as-is.
4. **Table rows** (`itinerary-table.tsx`): `SortableRow`'s `<tr>` → `motion.tr` with
   `rowEnterExit`; the per-day-group row map wrapped in `<AnimatePresence initial={false}>`
   (existing rows don't fade on load; new rows fade in, deleted rows fade out). Exit
   propagates through the `memo` wrapper via framer's PresenceContext.
   **Critical constraint:** rows animate **opacity ONLY** — never `y`/`transform`,
   never `layout`. The `<tr>` carries dnd-kit's drag-shuffle `transform`/`transition`
   inline; if framer animated `y` it would take permanent ownership of `transform` and
   clobber dnd-kit's drag transforms. Reorder stays 100% dnd-kit's job.

**Verification.** `npx tsc --noEmit` + `npm run build` clean. e2e green (21 tests):
drag-reorder, edit-interruption, smoke, route-hover, marker-hover, map-bugs — the
riskiest being drag-reorder (motion.tr × dnd-kit) and "markers present after reload".
Reduced-motion honored two ways (CSS @media + MotionConfig).
